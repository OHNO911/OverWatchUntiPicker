# OW Counter Lab 品質レビュー（2026-03-25）

## 総合評価
- **7.6 / 10**

## セクション別評価

### 1) アーキテクチャ / 責務分離: 8.5 / 10
- `core`（状態・計算・外部データ取得）と `ui`（描画）が分離され、保守しやすい構成。
- `main.js` がオーケストレーターとして機能し、処理フローも追いやすい。

### 2) ロジック品質（スコア計算）: 7.5 / 10
- `calculateAntiScores` は副作用がなく読みやすい。
- 一方で、重みが定数直書きで、妥当性検証や調整履歴がコード上で追いづらい。

### 3) データ品質（ヒーローデータ）: 7.0 / 10
- データが一元化され、型コメントもある。
- ただし巨大な静的配列のため、検証・更新ミス検出の仕組み（スキーマ検証）がない。

### 4) UI/UX: 8.0 / 10
- 視認性・情報密度・編集UI（Undo/Redo）が良い。
- ただし「なぜその推奨か」の説明粒度はまだ限定的で、初心者向けの理解負荷がある。

### 5) 信頼性 / 耐障害性: 7.0 / 10
- 取得失敗時にフォールバックしてクラッシュ回避できている。
- fetch エラー時のユーザー通知や再試行制御が薄く、観測性も弱い。

### 6) パフォーマンス: 7.5 / 10
- 規模に対して十分高速。キャッシュ戦略も妥当。
- ただし大規模データ化した場合は再描画単位の最適化余地あり。

### 7) テスト / 開発体験: 5.5 / 10
- README に手動確認手順があり、最低限の運用は可能。
- 自動テスト未整備で、仕様変更時の回帰検知が弱い。

## 10点にするための優先アドバイス（セクション別）

### 1) アーキテクチャ / 責務分離
1. `main.js` の window 公開関数群を `actions/` に分割して責務明確化。
2. 状態遷移をイベント駆動（`dispatch(action)`）に統一し、更新経路を一元化。
3. 設定・永続化・外部I/Oの境界をインターフェース化（将来テスト容易化）。

### 2) ロジック品質（スコア計算）
1. 重み（`*5`, `*8`, tank倍率など）を設定オブジェクトへ抽出。
2. スコア内訳（構成、個別、マップ）を構造化して返す。
3. 入力バリデーション（欠損 `matchups/antis`）と境界テストを追加。

### 3) データ品質（ヒーローデータ）
1. JSON Schema（または Zod など）で起動時バリデーション。
2. ID重複・参照不整合（`antis` 名称ミスマッチ）をCIで検出。
3. データ更新手順（追加・変更・削除）を README に規約化。

### 4) UI/UX
1. 推奨上位に「理由サマリ」を表示（例: 対ダイブ +8 / 対タンク +6）。
2. 結果カードに「詳細トグル」で内訳可視化（初心者と上級者を両立）。
3. 検索/フィルタ（ロール・アーキタイプ）を追加し操作時間を短縮。

### 5) 信頼性 / 耐障害性
1. マップ統計取得失敗時に UI 通知（トースト）を表示。
2. fetch のタイムアウト・再試行・バックオフを導入。
3. キャッシュ破損時の自己修復（不正JSON検知で破棄）を明示。

### 6) パフォーマンス
1. 描画差分更新（全文字列再生成から部分更新へ）を段階導入。
2. 重い計算を必要時のみ実行（入力差分の検出）。
3. 画像読み込み最適化（遅延読み込み、エラーフォールバック）を追加。

### 7) テスト / 開発体験
1. `core/calc.js` のユニットテストを最優先で追加。
2. `core/state.js` の undo/redo・制約適用の境界テストを追加。
3. CI で `lint + test` を必須化し、回帰を自動検知。
